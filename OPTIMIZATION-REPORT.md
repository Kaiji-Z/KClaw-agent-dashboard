# Kaiji Dashboard 优化报告

> 生成时间: 2026-04-11 15:00

## 🔍 问题诊断

### 当前架构：HTTP 轮询

```
浏览器 ──(每1.5s)──> Dashboard Server ──(N+1)──> Gateway API
     setInterval           HTTP proxy
```

**每次 `refreshData()` 的请求链路：**

| 步骤 | 请求 | 目标 |
|------|------|------|
| 1 | `sessions_list` | Gateway → 列出所有活跃会话 |
| 2-N | `sessions_history` × 每个Agent | Gateway → 获取每个Agent最近消息 |

**当前负载（5个Agent）：**
- 每 1.5 秒 = **6 次 HTTP 请求**
- 每分钟 = **240 次请求**
- 每小时 = **14,400 次请求**

**资源消耗：**
- Gateway CPU 持续 45-50%（已有记录）
- Dashboard Node.js 进程持续处理代理请求
- 浏览器频繁 GC（每次请求创建 Promise、解析 JSON、重渲染 DOM）
- 网络连接反复建立/关闭

---

## 💡 优化方案

### 方案 A：Server-Sent Events（SSE）— ⭐ 推荐

**原理：** 服务端有数据变化时主动推送，而非客户端反复询问。

```
浏览器 <──(事件流)── Dashboard Server ──(长轮询/SSE)──> Gateway
     EventSource        单连接复用
```

**改动：**
1. `server.js` 新增 `/api/events` SSE 端点
2. 服务端定期（如 10s）查询 Gateway，有变化才推送
3. 前端用 `EventSource` 替代 `setInterval`

**优势：**
- 请求量从 240/min → **6/min**（仅服务端查询）
- 浏览器零轮询，CPU 大幅降低
- 实现简单，纯 HTTP，无需额外依赖
- 自动重连机制内置

**劣势：**
- Gateway 本身没有 SSE API，需要 Dashboard 做中间层轮询
- 单向通信（对本场景够用）

---

### 方案 B：智能轮询 — 🥈 简单快速

**原理：** 保留轮询，但大幅减少无效请求。

| 优化项 | 改动 | 效果 |
|--------|------|------|
| **Page Visibility** | 标签页不可见时暂停轮询 | 切换标签省 90%+ 请求 |
| **自适应间隔** | 有活跃 Agent → 3s，全部空闲 → 30s | 空闲时请求量 ↓90% |
| **N+1 → 批量** | 合并 `sessions_list`+`history` 为单次请求 | 请求次数从 6→1 |
| **防抖渲染** | 多次更新合并为一次 DOM 更新 | 减少 GC 和重绘 |
| **条件历史查询** | 仅对最近活跃的 Agent 查 history | 跳过空闲 Agent |

**效果预估：**
- 正常使用：240/min → **~20/min**
- 标签页后台：240/min → **0**

**优势：**
- 改动量小，不需要新协议
- 向后兼容
- 立竿见影

**劣势：**
- 仍有轮询开销（虽然大幅减少）

---

### 方案 C：混合方案 — 🥇 最佳效果

**SSE（服务端推送） + 智能轮询（兜底）**

```
┌─ 正常模式：SSE 推送（服务端 10s 查一次，有变化才推送）
│
├─ 标签页隐藏：自动降级为 30s 长间隔轮询
│
└─ SSE 断开：自动 fallback 到 5s 轮询，恢复后切回 SSE
```

## 📊 方案对比

| 维度 | A: SSE | B: 智能轮询 | C: 混合 |
|------|--------|------------|---------|
| **请求量降低** | 97% | 90% | 99% |
| **实现复杂度** | 中 | 低 | 中高 |
| **实时性** | 秒级 | 秒级 | 秒级 |
| **改代码量** | ~150行 | ~50行 | ~200行 |
| **推荐场景** | 追求极低资源 | 快速修复 | 长期方案 |
